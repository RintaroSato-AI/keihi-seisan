# レシート経費精算アプリ 要件定義書

**バージョン**: 1.0（ドラフト）
**作成日**: 2026-07-14
**プロジェクト名**: keihi-seisan

---

## 1. システム概要

スマホで撮影したレシート写真から、AI が店名・日付・金額・品目を自動読み取りし、経費精算書（PDF/Excel）を作成する個人利用向け Web アプリケーション。

### 利用シナリオ
1. スマホのブラウザでアプリにアクセスし、レシートを撮影 or 選択してアップロード
2. AI（Claude Vision）が店名・日付・金額・品目を自動読み取り、費目（交通費・接待費など）も自動分類
3. 読み取り結果を画面上で確認・修正
4. レシートのない経費（電車賃など）は手入力で追加登録
5. 案件・プロジェクト単位で経費をまとめ、月次などのタイミングで経費精算書（PDF/Excel）を出力

---

## 2. 技術スタック

naishi-mitsumori（内装工事見積もりシステム）と同一構成を踏襲し、環境構築コストを抑える。

| 分類 | 技術 |
|------|------|
| フロントエンド | Next.js (App Router) / React / TypeScript |
| スタイリング | Tailwind CSS |
| バックエンド | Next.js API Routes |
| ORM | Prisma |
| DB | SQLite（ローカルファイル、`file:./prisma/dev.db`） |
| AI（画像読み取り・分類） | Anthropic Claude API（Vision） |
| PDF 生成 | @react-pdf/renderer |
| Excel/CSV 出力 | 別途ライブラリ選定（例: exceljs） |
| ファイルストレージ | ローカルファイルシステム（自分の Mac） |
| デプロイ | なし（ローカルサーバー起動のみ） |

### ⚠️ 運用上の注意点
- レシート画像はローカル（Mac）に保存する一方、撮影・登録はスマホ中心を想定している。
- そのため運用は「Mac でアプリのサーバーを起動 → 同一 Wi-Fi 内のスマホのブラウザからアクセス」という形になる。
- 外出先や別回線からはアクセスできない。将来的に外出先からも使いたい場合はクラウドストレージ＋デプロイへの切り替えが必要（今回は対象外）。

## 3. 機能一覧

### 3-1. レシート登録（写真あり）
- スマホカメラ起動 or 画像ファイル選択でレシート写真をアップロード
- Claude Vision API で以下を自動読み取り
  - 店名
  - 日付
  - 合計金額
  - 品目（可能な範囲で）
- 読み取り結果から費目（交通費・接待交際費・消耗品費など）を AI が自動分類
- 読み取り結果の確認・修正画面（AI 誤読の手動修正に対応）
- 案件・プロジェクトを選択して紐付け

### 3-2. レシート登録（写真なし・手入力）
- 電車賃・駐車場代などレシートがない経費を手入力で登録
- 日付・金額・費目・案件・備考を入力

### 3-3. 案件・プロジェクト管理
- このアプリ内で独立した案件・プロジェクトリストを管理（naishi-mitsumori とは連携しない）
- 案件の新規作成・一覧・選択

### 3-4. 経費精算書の作成・出力
- 案件・プロジェクト単位で経費をまとめて精算書を作成
- PDF 出力（レシート画像添付、費目別・合計金額を表示）
- Excel/CSV 出力（経理システムへの取り込み・集計用）

### 3-5. 一覧・検索
- 登録済みレシート・経費の一覧表示
- 案件・費目・期間での絞り込み

## 4. データモデル（案）

```
Project（案件）
  - id
  - name（案件名）
  - createdAt

Expense（経費）
  - id
  - projectId（→ Project）
  - date（日付）
  - storeName（店名、手入力の場合は null 可）
  - amount（金額）
  - category（費目：交通費・接待交際費・消耗品費 など）
  - receiptImagePath（レシート画像のローカルパス、手入力の場合は null）
  - memo（備考）
  - createdAt / updatedAt

ExpenseReport（精算書）
  - id
  - projectId（→ Project）
  - title
  - totalAmount
  - generatedAt
  - pdfPath（生成済み PDF のパス）
```

※ インボイス制度対応（登録番号・税率区分の記録）は対象外。簡易的な社内精算を目的とする。

---

## 5. 非機能要件

- **利用者**: 個人利用のみ（Phase 1）。ログイン・認証機能は実装しない。
- **将来拡張**: 複数ユーザー対応・承認フロー（申請中／承認済み／却下のステータス管理）を見据え、`Expense` に `userId` を後付けしやすいスキーマにしておく。
- **データ保持期間**: 特に期限を設けず、削除機能は用意しつつ自動削除は行わない。
- **対応端末**: スマホ中心（レスポンシブ対応）。PC からの一覧確認・修正にも対応。

---

## 6. 対象外（Phase 1 では実装しない）

- インボイス制度（適格請求書等保存方式）への対応
- 複数ユーザーでの利用・ログイン機能
- 承認フロー（申請・承認・却下のステータス管理）
- naishi-mitsumori の案件データとの連携
- クラウドストレージ・外部からのリモートアクセス
